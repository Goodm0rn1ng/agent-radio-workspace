# ADR 0003：翻译策略——分批 + 段对齐 + 校验重试

**日期**：2026-05-15  
**状态**：采纳

## 背景

Whisper 输出是 segment-level 时间戳列表（每段 3-10 秒）。要保留双语时间戳对齐，翻译策略有三种：

1. **整篇翻译**：一次性扔给 LLM。简单但 LLM 会改变段落数量，时间戳全乱。
2. **逐段单独翻译**：一段一调用。时间戳天然对齐，但慢且没上下文，质量差。
3. **分批翻译，强约束输出段数对齐**：每批 N 段一起翻，强制要求"输出与输入相同数量的段落"。

## 决策

采用策略 3：每批 30 段，要求 JSON 输出 `[{"i": 0, "zh": "..."}, ...]`，解析后校验段数一致。

## 理由

- **时间戳对齐**：输入第 i 段 → 输出第 i 段对应的中文，时间戳沿用。
- **质量**：30 段 ≈ 2-3 分钟内容，有足够上下文，翻译质量接近整篇。
- **可恢复**：偶尔段数对不上，重试一次（temperature=0.2 → 0.0 更严格）。二次仍失败降级为单段翻译。
- **token 高效**：批量调用比逐段省 30-40% token（共享 system prompt）。

## LLM 选型

v1 翻译使用 **DeepSeek API**。当前配置为 `deepseek-v4-flash`：
- 中日翻译质量略低于 Claude，但价格便宜 ~10 倍
- 月度翻译成本可从 $0.5 降到 $0.05 量级
- 旧配置曾使用兼容别名 `deepseek-chat`；DeepSeek 官方已提示该别名后续会废弃，因此改用显式模型名。

若实际质量不达标（人名/术语经常翻错），切回 Claude Sonnet（改一行环境变量即可）。

## 后果

- 正面：时间戳精确对齐，成本最低。
- 负面：LLM 偶发段数错位需重试逻辑兜底，增加约 50 行代码。
