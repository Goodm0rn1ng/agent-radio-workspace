# 产品需求文档 (PRD)：Project "Radio-Oshikatsu" · 个人化广播内容自动化与知识库 Agent

**文档版本:** V2.0（2026-05-30 按已交付现实裁剪；前身 V1.1 为云端 Serverless 愿景稿）
**产品定位:** 面向单一运营者的、**单机自部署**的广播内容自动化流水线 + 可问答的专属知识库。从"录制日本声优广播"到"逐字稿翻译 / 摘要 / 推送 / 沉淀为知识图谱 / 带出处问答"全链路自动化，HITL 只卡在少数高风险节点。

> **V1.1 → V2.0 变更说明**：V1.1 描述的是"纯云端 Serverless + 实时流 + 云原生向量库 + 微信小程序生态"的理想架构，与实际交付不符。V2.0 把文档对齐到**真实已实现的系统**，并把未实现的宏大设想显式移入「非目标」与「未来可选」，以体现范围取舍。

---

## 1. 产品概述

### 1.1 问题与背景
追踪日本声优广播（如《羊宮妃那のこもれびじかん》）的爱好者，要消费一期节目需要：定时蹲守直播 → 录音 → 切片 → 日文转写 → 中文翻译 → 人工通读码字摘要 → 跨期考据（"这期填了上期的坑"）。整套流程单期耗时数小时，且无法规模化、无法检索历史。

### 1.2 产品愿景（裁剪后）
做一个**给个人用的、能长期无人值守运行的内容 Agent**：准点录制目标广播，自动产出双语逐字稿与结构化摘要，推送到 Telegram / 小红书供同好消费，并把每一期沉淀进**专属知识图谱**，支持带出处的自然语言问答与"主持人口吻来信回复"。

**不做**全云端 SaaS、不做实时流、不做多模态视频切片、不做小程序生态飞轮——这些在 V1.1 是愿景，在 V2.0 是明确的非目标（见 §3）。

### 1.3 目标用户
- **主运营者（=作者本人）**：部署、调度、审批、提问的唯一操作者。
- **同好消费侧**：通过 Telegram 频道 / 小红书被动消费产出的图文，不直接操作系统。

### 1.4 核心价值
- **降本**：单期内容生产周期从"人工数小时"压到"端到端分钟级、零人工码字"。
- **沉淀**：每期入库形成可跨期检索的饭圈知识库（图谱 + 向量），自动关联历史事件。
- **可信**：所有问答带【出处:期数+时间戳】，并以"防编造"硬约束兜底——宁可诚实弃答，不臆造。

## 2. 系统架构与技术栈（实际）

**部署形态**：单机（macOS 本地 / 单台东京区 VM 二选一），Radio 生产端 + radio_kg 消费端共一个端口（FastAPI 子应用挂载），调度 daemon 独立常驻。非云 Serverless。

```
[采集] radiko(API+playwright兜底) / YouTube Live / 本地视频 / 播放列表
   → [切片] ffmpeg
   → [STT]  Groq Whisper-large-v3（动态 prompt 注入术语）
   → [翻译] Deepseek（默认）/ Claude Haiku（精翻可选）
   → [名词·术语修正] 自定义词典
   → [摘要] Gemini 2.5 Flash（注入往期记忆 + 常驻环节库）
   → [推送] Telegram + 小红书
   → [交接] POST /api/ingest 给 radio_kg
                 ↓
[知识库] DocAgent→Extractor→Inspector→Sync 入图谱(Neo4j) + 向量库(Chroma)
   → GraphRAG 两段式混合检索（向量+图谱，RRF 融合，结构化带出处答案）
   → 问答 / 统计聚合 / 实体档案 / 主持人「来信模式」/ 自进化订正
```

| 层 | 选型（实际） |
| :-- | :-- |
| 编排 | 普通 async 流水线 + 单进程串行队列（非 LangGraph 状态机）；知识库侧用 LangGraph 编排入库与问答图 |
| 流媒体 | `ffmpeg` + `yt-dlp` + `playwright`（radiko 录制兜底） |
| STT | Groq Whisper-large-v3（批处理离线文件，非实时流） |
| 翻译 | Deepseek `deepseek-v4-flash`（主）+ Anthropic Claude Haiku（精翻可选） |
| 摘要 | Google Gemini 2.5 Flash |
| 存储/记忆 | 本地 Neo4j（图谱）+ Chroma（e5-base 多语向量），通过官方 **MCP**（mcp-neo4j-cypher）访问 |
| 交互/审批 | Telegram Bot（HITL Inline Buttons） |
| 调度 | APScheduler daemon（sqlite jobstore） |
| 工程 | pydantic Settings（SecretStr 管密钥）、指数退避重试、metrics.jsonl 埋点、GitHub Actions CI（ruff + pytest） |

## 3. 范围与非目标（Scope & Non-Goals）

### 3.1 本期范围（已交付）
采集（多源）、批处理转写翻译摘要、名词/术语修正、双语逐字稿、结构化摘要、Telegram/小红书推送、HITL 新环节审批、定时调度、自动入库知识库、GraphRAG 问答、统计/实体档案、主持人来信模式、自进化订正、期数放送日期问答。

### 3.2 非目标（从 V1.1 砍掉，并说明理由）
| 砍掉的设想 | 理由 |
| :-- | :-- |
| 云端 Serverless 部署（腾讯云 SCF / 阿里云 FC / CloudRun） | 录制是"常驻 + 长跑 + 地域锁 + 有状态 + 重二进制(playwright)"工作负载，与请求驱动、有时长上限的 FaaS 根本冲突；个人项目重性价比，单机/单 VM 足够 |
| Deepgram 实时流转录 | "播完数分钟内出稿"已满足个人需求；实时流大幅增加复杂度与成本，ROI 低 |
| 云原生向量库 / Qdrant Cloud | 数据量千级实体，本地 Chroma + Neo4j 零成本可控；已用 MCP 抽象，未来要换不被锁死 |
| 微信小程序内容引擎 / 生态飞轮 | 聚焦"自用内容提炼 + 知识沉淀 + 问答"，不做分发生态扩张 |
| 多模态情绪高光视听切片（声波纹视频） | 投入产出比低，核心价值在文本提炼与知识库 |
| 公开/商业化分发 | 广播版权、声优声音权、平台 ToS 未解；明确限定在**个人自用**范围内 |

## 4. 核心功能需求（对齐实现）

### 4.1 数据接入（多源）
- **定时直播录制**：APScheduler 按日本节目档期准点拉取 radiko 音频流（API 源 + playwright 源兜底）；录制前做 radiko 区域/可达性健康探测，失败发 Telegram 告警。
- **按需音视频提取**：提交 URL（YouTube Live / 视频）或本地文件，提取音频进入同一处理链路。
- 成功后清理切片释放空间；**失败保留切片与中间产物**便于复盘。

### 4.2 处理与记忆
- **专有名词映射**：STT 后接自定义词典 + 术语库，强制纠正声优昵称 / Live 名 / 企划术语。
- **往期记忆注入摘要**：摘要阶段注入 `history_context.jsonl` 往期 key_topics 与常驻环节库，实现跨期关联。
- **知识图谱沉淀**：消费端把每期解析为带时序与出处的实体-关系三元组，CDC 增量更新（旧事实保留为历史线）。
- **时间加权 / 两段式检索**：先粗排到相关期，再细取时间窗口；向量 + 图谱双路 RRF 融合。

### 4.3 内容生成与人机协同（HITL）
- **风格化摘要**：分段、标题化、保留现场氛围标签。
- **HITL 只卡两处高风险点**：① 摘要发现的**新环节**入库前需审批（不直接写常驻环节库）；② 知识库**知识冲突**需人工裁决。其余全自动。
- Telegram Inline Buttons：确认 / 重写 / 忽略。

### 4.4 知识库问答（消费端）
- 带【出处】的问答、统计聚合、单实体完整档案、主持人口吻"来信模式"（中日双语）、自然语言订正知识库（"订正/记住"→预览后写入，以用户为准）。
- **防编造硬约束**：答案为"事实 + source_id"结构化产出并逐条校验，无依据则弃答。

## 5. 非功能需求（现实化）

- **性能**：按需视频（≤15min）从提交到出稿目标 ≤ 数分钟；直播结束后数分钟内推送摘要。知识库侧 NFR 已验证：MCP 单跳查询 p95 < 200ms、混合检索 < 2s。
- **可靠性**：外部 API 调用 3 次指数退避重试；pipeline 失败发 Telegram；调度 daemon 崩溃自动重启（restart 策略 / LaunchAgent）。
- **网络约束**：录制端必须运行于**日本 IP**（radiko 地域锁），其余处理/问答无地域要求——二者可拆分部署。
- **成本**：API-first（Groq/Deepseek/Gemini），单机基础设施近零成本；metrics.jsonl 记录每期 token 用量用于成本核算。
- **数据安全**：密钥以 SecretStr/环境注入；知识图谱定期备份（不可再生资产）。

## 7. 关键产品决策与权衡（Key Decisions）

1. **单机自部署 ＞ 云端 Serverless**：基于工作负载本质（常驻/长跑/地域锁/有状态/重二进制）与个人项目成本，主动放弃 V1.1 的云愿景，换取可运维性与零成本。
2. **批处理 STT ＞ 实时流**：以"够用"为准绳砍掉 Deepgram 实时，控复杂度。
3. **HITL 收敛到两个高风险点**：在自动化率与正确性之间取平衡——只在"新环节登记"和"知识冲突"处要人，其余放手。
4. **防编造硬约束**：明确以牺牲部分回答完整度换取可信度，这是 RAG 产品的信任底线。
5. **生产者/消费者用 `03/04/05.json` 数据契约解耦**：两系统可独立演进、独立部署（也支撑"日本 IP 录制端 + 大内存处理端"拆分）。
6. **走官方 MCP 访问存储**：保留未来替换图/向量后端的空间，不被本地选型锁死。

## 8. 演进路线图（Roadmap）

### 已交付
- **Phase 1 链路打通**：URL/文件 → 提取 → STT → 翻译 → 摘要 → Telegram。
- **Phase 2 记忆与审批**：向量库/图谱 Embedding 与检索、Bot HITL 闭环、名词纠错词典。
- **Phase 3 直播监控与图谱问答**：定时录制引擎攻克 radiko；GraphRAG 两段式检索、统计/档案/来信模式、自进化订正、期数元数据问答。

### 未来可选（非承诺）
- 上云（优先东京区单 VM / 拆分录制端与处理端，详见部署 runbook）；漏录看门狗 + timeshift 自动补录。
- 外部边界（STT/翻译/各采集源）的测试覆盖与翻译质量评测闭环。
- 成本/延迟/失败率的聚合可观测面板。
- 若未来公开化：版权/ToS 合规评审、内容溯源——当前明确不在范围内。
