# 用户体验优化研究（2026-06-13）

基于当前真实运行状态的调研：`/api/health`、`/api/pending` 现场数据、bench_perf 实测、QA 链路代码审查。
配套的性能优化（检索 769ms→408ms、e5 模型去重）已落地，见 `radio_kg/CHANGELOG.md`。

## 一、现有流程盘点

| 流程 | 入口 | 现状 |
|---|---|---|
| 智能问答 | `/chat`（SSE 分阶段进度、👍/👎 反馈） | 单问 ~20s，3 次串行 LLM 调用 |
| 录制→入库 | 调度 daemon + `/radio` | 自动化完整，失败靠日志发现 |
| 入库审批 | `/dashboard` 待处理卡片 | **积压 15 线程 / 27 项，最早 5/30** |
| 直播切片 | `/clipper` + Telegram 按钮 | 两段式审核已上线；>50MB 只回本地路径 |

## 二、发现的问题（按影响排序）

### P0｜审批队列被 no-op 项淹没（操作负担 + 真冲突被埋没）
实测 `/api/pending`：27 项中 **26 项是 no-op**——`suggested_name == original_name`（Inspector「无法确定修正目标」也推成人工审批），积压两周。唯一一条真实冲突（#51 期『担当する』环节由「ウーシャンのゆり色パレット」变更为「週刊こもりすプチニュース」，疑似真实节目改版）被淹没在里面。
**这与 PRD 3.3 的设计相悖**：PRD 写明「无法判断 → 保留原三元组 + 低置信标记，供人工抽检」，不应触发 interrupt。
建议：
1. InspectorAgent：`suggested_name == original_name` 的 `llm_audit` 项不再触发 interrupt，按低置信标记直接入库（与 PRD 对齐）；
2. 审批 UI 增加「全部忽略 / 按类型批量处理」；
3. 一次性清理现有积压（26 条 no-op 自动 resolve，留 1 条真冲突人工裁决）。

### P1｜问答 20 秒延迟，瓶颈在 3 次串行 LLM 调用
实测 `/api/ask2` 20.0s；检索仅 0.4s，其余为 analyze → answer_structured → verify 三连 LLM。建议（按性价比）：
1. **analyze 降级**：anchor/期号抽取大部分可由已有正则（`_filters`）+ 词典覆盖，LLM 只兜底——可省 3–5s；
2. **答案缓存**：同一问题（归一化后）在索引版本不变时直接复用，重复提问秒回（`index_version.py` 的指纹现成可作缓存键）；
3. SSE 进度条加「预计还需 Ns」（各阶段历史耗时已有 trace 日志可统计）。

### P1｜失败与异常对用户不可见（要主动查才知道）
`/api/health` 能报 degraded、`ingest_commit_log` 记录了 **15 个非终态 ingest**，但没有任何主动通知。录制失败/入库挂起/组件退化，用户要么翻日志要么下次用时才发现。
建议：复用现成 Telegram bot 通道做主动告警——录制失败、ingest 非终态超 1 小时、health degraded 时推一条消息（含一键跳转 dashboard 链接）。这是把已有数据接到已有通道，改动小收益大。

### P2｜Telegram 切片 >50MB 死路
现行为：回「已切出完整片段 + 本地路径」——手机上的用户拿不到文件。本地 server 已有 HTTP Range 视频能力（`/api/slice/preview_video`）。
建议：>50MB 时回一条**可点开的本地网络播放链接**（同 Wi-Fi 可看可存），并附可选「压缩到 50MB 内再发」按钮，把决定权给用户而不是直接放弃。

### P2｜出处只是文本，无法一键回听
PRD 4.2 规划了「点击出处 → 期数+时间戳气泡」，当前答案里出处是纯文本（如 `【出处:第2期 00:01:30】`）。源音频就在 `Radio/data/recordings/`，server 能伺服媒体。
建议：出处渲染成链接，点击在页内拉起 `<audio>` 定位到 start_time 播放 ±30s——「事实溯源」从『可读』升级为『可验』，是本产品差异化卖点的最后一公里。

## 三、内容准确性专项

1. **verifier 误杀**（已知 trade-off：completeness 0.71→0.50）：对「概括/改写」类事实再放宽一档——verify 不通过的事实不直接丢弃，降级为「（待核实）」尾注输出，把裁决权交给用户而非静默吞掉。
2. **专属 stt_prompt 覆盖面**：幻听治理（人名词典 prompt）目前只有 `minetsuki_ritsu` 方案配了专属 prompt，其他节目仍用全局 prompt，存在异节目人名互染风险。建议为每个活跃节目方案补 `processing.stt_prompt`。
3. **图谱存量噪音**：42 个孤儿实体（峰月律首轮污染清理残留）+ graph/vector 的 episode_label 命名不齐（`/api/index_status` 已持续报漂移）。建议跑一次 `repair_graph` + label 规范化，让漂移检测归零、不再「狼来了」。
4. **评估回归化**：`eval_qa20` 目前是手动跑。建议每次入库/改 prompt 后自动跑并与上次对比，三指标跌幅超阈值即 Telegram 告警——让「内容准确性」有持续监控而非一次性测量。

## 四、建议落地顺序

| 顺位 | 事项 | 改动量 | 收益 |
|---|---|---|---|
| 1 | Inspector no-op 不再 interrupt + 清积压 | 小 | 审批队列即刻可用 |
| 2 | Telegram 主动告警（失败/挂起/degraded） | 小 | 异常从「被发现」变「被通知」 |
| 3 | 答案缓存 + analyze 正则前置 | 中 | 重复问秒回、平均延迟显著降 |
| 4 | 出处可点击回听 | 中 | 产品差异化卖点闭环 |
| 5 | verifier 降级输出 +（待核实）尾注 | 小 | 完整性回升不牺牲可信度 |
| 6 | eval_qa20 回归化 | 中 | 准确性持续可观测 |
