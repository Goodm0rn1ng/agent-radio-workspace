# PRD：基于多 Agent 协同的广播音频深度知识管理与智能问答系统

## 1. 项目概述与核心目标

### 1.1 产品定位
一款基于多 Agent 协同的广播/音频内容深度知识管理与智能问答系统。系统通过构建"广播时序知识图谱"，将碎片化、口语化的长音频对话数据转化为结构化的硬核知识库，利用大模型在处理复杂人物关系、时间轴和深度多跳推理上的能力，实现精准的图向量混合 RAG 问答与事实溯源。

### 1.2 核心痛点与解决方案

- **痛点 1：口语化严重与信息稀释。** 广播内容信息密度低、跨期数长，传统 RAG（检索增强生成）容易因为文本切片断章取义而检索失败。
- **痛点 2：高昂的维护与重算成本。** 广播具有持续更新（如周更）的特点，每次引入新内容都进行全量知识库重构会造成极大的 Token 和算力浪费。
- **痛点 3：幻觉与无法溯源。** 大模型在回答复杂的长周期事件时极易产生幻觉，且无法给出让用户信任的具体出处。
- **解决方案：**
  - 引入 GraphRAG（图增强检索），将人名、企划、生草梗（Inside Jokes）固化为图谱节点，通过边（Edges）关联期数与时间线。
  - 采用 LangGraph 构建带有人机协同（Human-in-the-loop）的 CDC (变更数据捕获) 增量更新机制。
  - 全面接入 MCP (Model Context Protocol) 行业标准协议，解耦存储与计算。
  - 提供端到端的音频模态时序溯源 [期数 + 时间戳] 机制。

## 2. 系统全局架构与技术选型

系统整体从传统的线性流水线升级为基于图状态机的网状协同架构，底层存储与计算工具全面实现标准化接口封装。

```
                    ┌────────────────────────┐
                    │       用户前端 UI       │
                    └───────────┬────────────┘
                                │ (1. 提问 / 2. 上传文档)
                                ▼
              ┌───────────────────────────────────┐
              │    LangGraph 主控状态机 (Master)    │
              └─────────────────┬─────────────────┘
                                │
        ┌───────────────┬───────────────┬───────────────┐
        ▼               ▼               ▼               ▼
 ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
 │  文档解析    │ │  知识抽取    │ │  审核纠偏    │ │  知识更新    │
 │ (DocAgent)   │ │ (Extractor)  │ │ (Inspector)  │ │ (SyncAgent)  │
 └──────┬───────┘ └──────┬───────┘ └──────┬───────┘ └──────┬───────┘
        │                │                │                │
        └───────────────┬┴────────────────┴────────────────┘
                        │ (基于 MCP 协议读写)
                        ▼
         ┌─────────────────────────────────────┐
         │       MCP 统一数据访问层             │
         ├──────────────────┬──────────────────┤
         │ Neo4j MCP Server │ Pinecone/Vectara │
         └────────┬─────────┴────────┬─────────┘
                  │                  │
                  ▼                  ▼
             [图数据库存储]      [向量数据库存储]
                  │                  │
                  └────────┬─────────┘
                           │ (GraphRAG 混合检索)
                           ▼
                 ┌──────────────────┐
                 │    问答 Agent    │ ──► 答案 + [期数+时间戳]
                 └──────────────────┘
```

### 2.1 状态机引擎：LangGraph
- **状态化管理 (Stateful Agents)：** 利用 LangGraph 的 StateGraph 定义统一的共享状态（State），承载当前解析文本、抽取的临时三元组、置信度以及冲突标记。
- **断点与人机协同 (Interrupts)：** 当新旧数据发生逻辑冲突时，状态机触发 `interrupt()` 自动挂起，将冲突推送至前端审批流。等待用户确认后，利用 Checkpointer 从断点恢复（Time Travel）继续执行。

### 2.2 统一接口规范：MCP (Model Context Protocol)
系统拒绝使用任何私有 API 对接存储层，统一采用 JSON-RPC 2.0 的 MCP 协议。

- **集成市面成熟 MCP 服务：**
  1. **Neo4j 官方 MCP Server (mcp-neo4j-memory)：** 提供标准的节点搜索、关联查找及受控的 Cypher 语句执行。具备实体安全隔离，防止 Agent 动态生成查询时引发注入攻击。
  2. **Vectara / Pinecone MCP Server：** 用于高性能向量相似度匹配及标准化 RAG 检索管道接入。

### 2.3 混合存储层 (Hybrid Storage)
- **向量数据库 (Vector DB)：** 存储原始对话的 Embedding 切片，用于处理模糊的语义搜索、情绪搜索。
- **图数据库 (Graph DB - Neo4j)：** 存储结构化的实体、属性和关系，用于处理绝对精确的人物关系和跨期数时序推理。

## 3. Agent 核心架构与职责划分

### 3.1 文档解析 Agent (DocAgent)
- **输入：** 原始文本、音频（集成 ASR 后结构化文本）、同参演人员的其他广播/杂志 PDF / 百科网页。
- **核心职责：**
  - **时序对齐：** 自动识别或引导用户输入当前文档的"录制时间"或"发布期数"，将其作为元数据（Metadata）硬编码进每一段文本 Chunk。
  - **角色纠偏与清洗：** 清洗 ASR 文本，修正多人对话中由于叠词、插话导致的角色错乱，输出标准的 `[时间戳] 说话人: 内容` 结构化文本。

### 3.2 知识抽取 Agent (ExtractorAgent)
- **核心职责：** 从清洗后的文本中提取 实体（Entities）、属性（Attributes） 和 关系（Relations），并绑定时序元数据。
- **消歧机制（Disambiguation）：** 广播中经常出现外号、缩写（如"Leader"、"小张"）。该 Agent 必须调用 MCP 查询已有知识库进行实体对齐（Entity Linking），识别出"小张"的核心实体 ID。

### 3.3 审核 Agent (InspectorAgent)
- **角色定位：** 作为知识入库前的最后一环，位于 ExtractorAgent（知识抽取）与 SyncAgent（知识更新）之间。其核心职责是拦截并修正由 ASR 错误、口语化简称或大模型幻觉导致的“脏数据”，确保录入图谱的事实具备绝对准确性。
- **典型场景：** 当 ExtractorAgent 从转写文本中抽取出 `[羊宮妃那]-(所属事务所)->[青鬼プロダクション]` 时，InspectorAgent 需要识别这很可能是“青二プロダクション”的同音/近音误抓，而不是新的真实事实。
- **处理结果：**
  - **自动纠偏成功：** 对高置信纠错结果直接改写临时三元组，并记录审核说明，顺延至 SyncAgent 写入。
  - **高风险但不确定：** 触发 LangGraph `interrupt()`，推送至人机协同审批流，由用户确认“采用纠偏 / 保留原文 / 忽略该事实”。
  - **无法判断：** 保留原三元组，但附加低置信风险标记，供后续人工抽检。

#### 3.3.1 三层纠偏防御策略
- **行业知识基座（Domain Vocabulary Matching）：** 系统预先加载“ACGN/声优行业核心专有名词词典”，包含日本主流声优事务所与高频专有名词，如 `青二プロダクション`、`大沢事務所`、`Arts Vision` 等。当审核 Agent 看到 `青鬼プロダクション` 时，通过 Jaro-Winkler、编辑距离或等价字符串相似度发现其与 `青二プロダクション` 高度接近，触发预警。
- **日语读音发音相似度（Phonetic Distance / Kana Level Verification）：** 将提取实体转化为平假名/片假名形态或利用词典内置读音别名进行比对。例如 `青二` 对应 `あおに`，`青鬼` 可能被识别为 `あおき` 或 `あおおに`。若读音只有一字之差，且错误候选在行业知识中不存在，则判定为 ASR 抓取错误。
- **历史图谱频次校验（Graph Frequency Check）：** 审核 Agent 通过 MCP 查询 Neo4j 历史数据，检查该主体或相关人物在历史广播中被连接到的组织实体。若前 27 期均指向 `青二プロダクション`，本期孤立出现一次 `青鬼プロダクション`，系统将其视为孤立噪声，自动纠偏或挂起。

### 3.4 知识更新 Agent (SyncAgent)
- **技术核心：** CDC (Change Data Capture) 增量更新机制。
- **核心职责：**
  - **图谱剪枝与版本控制：** 当广播中出现关系变更（如：某个企划负责人由 A 变更为 B），系统需断开旧边，新建新边，并为边赋予时序属性（start_epoch, end_epoch），保留历史变更线。
  - **冲突挂起：** 发现强冲突（如同名实体属性彻底矛盾）且无法自主置信度对齐时，向 LangGraph 状态机发送中断信号。

### 3.5 问答 Agent (Q&A Agent)
- **核心职责：** 接受用户自然语言提问，执行 GraphRAG 混合检索，将两路检索结果通过 Reranker 融合打分，生成带有引用出处的精准答案。

## 4. 功能需求详情 (Functional Requirements)

### 4.1 GraphRAG 混合检索功能
- **需求描述：** 系统在执行问答时，必须同时兼顾模糊语义和硬核拓扑关系。
- **执行流：**
  1. **向量支线：** 检索召回与问题语义相似的对话切片（解决"当时聊到这个话题时气氛如何"类问题）。
  2. **图谱支线：** 沿实体节点进行多跳拓扑检索（解决"声优 A 吐槽过的那个企划最终是由谁负责的"类跨期数复杂推理）。
  3. **融合层：** 使用重排模型（Reranker）对两路数据进行交叉拟合，提炼生成上下文（Context）输入大模型。
- **日文向量检索策略：**
  - 默认使用 `intfloat/multilingual-e5-base` 重建 Chroma 向量库，适配本地 16GB M2 的质量、内存与速度平衡。
  - `intfloat/multilingual-e5-large` 仅作为可选高质量模式；只有在 e5-base 跑完既有问答样例后仍出现明显召回不足时再启用。
  - 对跨语种问题与日文原文表达不一致的场景，Q&A Agent 需生成贴近日文转写文本的 `search_query`，并在语义召回之外叠加少量关键词召回兜底。

### 4.2 音频模态事实溯源（初步阶段）
- **需求描述：** 用户在使用问答功能时，系统生成的每一个核心事实性结论，都必须提供原始出处供用户查证。
- **用户侧交互 (UX/UI)：**
  - 在生成的答案文本中，凡是涉及图谱推理出来 Fact（事实）的地方，句末需提供一个"查看出处"的动态图标。
  - 用户悬停或点击该图标，弹出一个气泡卡片，标准格式显式为：`[广播名称/期数] + [具体时间戳/区间]`。
  - **示例效果：** "...该企划最终由李四接手负责 [出处:《XX广播》第 12 期 00:15:23 - 00:16:00]。"
- **开发侧数据链：**
  - DocAgent 产出的 Chunk 必须带有 `start_time` 和 `end_time`。
  - ExtractorAgent 抽取的实体关系，必须将当前期数和时间戳区间作为"边（Edge）"的属性写入 Neo4j。

### 4.3 增量更新与人机协同审批看板
- **需求描述：** 提供一个前端 Dashboard，处理 LangGraph 状态机中断抛出的冲突数据。
- **用户侧交互 (UX/UI)：**
  - 界面展示"待处理知识变更"卡片。
  - 对比显示：`【已有知识：企划 A 负责人 -> 张三】` vs `【新广播提取：企划 A 负责人 -> 李四】`。
  - 提供三个操作按钮：`[确认变更（保留历史线）]`、`[覆盖旧知识]`、`[忽略新提取]`。用户点击后，LangGraph 状态机恢复执行。

## 5. 新增核心模块：审核 Agent (InspectorAgent)

### 5.1 角色定位
作为知识入库前的最后一环，位于 ExtractorAgent（知识抽取）与 SyncAgent（知识更新）之间。其核心职责是拦截并修正由 ASR 错误、口语化简称或大模型幻觉导致的“脏数据”，确保录入图谱的事实具备绝对的准确性。

### 5.2 核心工作流与防错机制
```
┌──────────────────┐
│  ExtractorAgent  │ (抽取产生临时三元组)
└────────┬─────────┘
         │ [声优A] -> (所属事务所) -> [青鬼プロダクション]
         ▼
┌──────────────────┐
│  InspectorAgent  │ (审核与纠偏)
└────────┬─────────┘
         ├─► 1. 检索常驻词典 / 百科 (发现“青二”高频词)
         ├─► 2. 计算语音相似度 (あおに vs あおおに 极度相似)
         ├─► 3. 自动纠偏 / 标记高风险
         ▼
┌──────────────────┐
│    LangGraph     │ ──► [高风险冲突] ──► 触发 interrupt() 推送前端
│    状态机中心    │ ──► [自动纠偏成功] ──► 顺延至 SyncAgent 写入
└──────────────────┘
```

### 5.3 三层纠偏防御策略
- **行业知识基座（Domain Vocabulary Matching）**
  - **机制：** 系统预先加载一份“ACGN/声优行业核心专有名词词典”，包含日本主流声优事务所，如 `青二プロダクション`、`大沢事務所`、`Arts Vision` 等。
  - **动作：** 当审核 Agent 看到 `青鬼プロダクション` 时，比对词典发现其与 `青二プロダクション` 的 Jaro-Winkler 距离、编辑距离或字符串相似度极高，触发预警。
- **日语读音发音相似度（Phonetic Distance / Kana Level Verification）**
  - **机制：** 将提取出的实体转化为平假名/片假名形态进行注音比对，或使用词典维护的读音别名。
  - **实例：** `青二` -> `あおに`；`青鬼` -> `あおき` 或 `あおおに`。
  - **动作：** LLM 或规则审核器在日语上下文下判断两者的 ASR 混淆度。如果读音只有一字之差，且后者在行业常识中不存在，则判定为 ASR 抓取错误。
- **历史图谱频次校验（Graph Frequency Check）**
  - **机制：** 审核 Agent 通过 MCP 协议查询 Neo4j 历史数据：“在之前的广播中，该声优或相关人员提及过什么事务所？”
  - **动作：** 如果历史图谱中稳定出现 `青二プロダクション`，本期突然出现一次 `青鬼プロダクション`，系统判定此为孤立噪声，自动修正或挂起。

### 5.4 存量图谱修复策略
- **适用范围：** InspectorAgent 负责阻断新脏数据；对在 InspectorAgent 上线前已经写入 Neo4j 的脏实体，需要提供受控的存量修复入口。
- **实体归并：** 将已确认的 ASR 错误实体（如 `青鬼プロダクション`）重定向到标准实体（`青二プロダクション`），迁移该节点的所有入边/出边，并保留边上的期数、时间戳、置信度与出处元数据。
- **安全边界：** 存量修复仍只能调用项目内固定、参数化的 GraphStore 模板；不得由 LLM 生成 Cypher。
- **验证要求：** 修复后重新执行相关问答样例，确认图谱检索、向量检索和最终答案均不再出现错误实体。

## 6. 关键数据流向 (Data Flow)

### 6.1 文档输入与增量入库数据流
```
[新期音频/文本]
       │
       ▼
1. DocAgent 解析 ──► 产出带时间戳清洗文本 (Metadata: Episode, Timestamp)
       │
       ▼
2. ExtractorAgent ──► 抽取 [实体-关系-实体] 三元组
       │
       ▼
3. InspectorAgent ──► 词典 / 读音 / 历史图谱三层审核
       │
       ├──► [自动纠偏成功] ──► 修正临时三元组
       │
       └──► [高风险不确定] ──► LangGraph 触发 interrupt() ──► 用户审批
       │
       ▼
4. SyncAgent 对比 ──► 调用 Neo4j MCP 检查是否存在冲突
       │
       ├──► [无冲突] ──► 通过 MCP 写入 Neo4j(图) & Pinecone(向量)
       │
       └──► [有冲突] ──► LangGraph 触发 interrupt() ──► 挂起并推送前端审批 ──► 用户确认 ──► 恢复写入
```

## 7. 非功能性需求与安全边界 (Non-Functional Requirements)

### 7.1 性能指标 (Performance)
- **MCP 传输延迟：** 主控与 MCP Server 间通过 JSON-RPC 传输的工具调用单次延迟需 < 200ms。
- **混合检索响应时间：** GraphRAG 混合检索（向量检索 + 图拓扑 2 跳查询 + Reranker 重排）的总耗时控制在 2 秒以内（不含 LLM 生成文本时间）。

### 7.2 边界情况处理 (Edge Cases)
- **无时间戳的外部材料：** 当用户上传百科网页或杂志 PDF 等无音频时间戳的材料时，DocAgent 自动将时间戳元数据降级替代为 `[文件名/文献名] + [第 X 页 / 第 X 段]`，确保溯源链路字段完整，不触发系统报错。
- **极度口语化消歧失败：** 若遇到"那个人今天也来了"等无法解析的模糊代词，ExtractorAgent 需自动放弃该关系的建立，做丢弃处理，防止垃圾数据污染图谱。

### 7.3 安全边界 (Security)
- **图谱防注入攻击：** 严禁大模型直接拼接原生 Cypher 语句。所有对图数据库的读写必须严格限制在 Neo4j MCP Server 暴露出的预定义工具集（如 `search_nodes`, `merge_directed_relationship`）范围内，实现底层的语义层（Semantic Layer）隔离。
